# Handover Note RAG Search System

[日本語](README.md) | **English**

---

## Overview

A fully local RAG (Retrieval-Augmented Generation) search system for handover note CSV data from a water treatment facility.  
Like [NotebookLM](https://notebooklm.google.com/), you ask questions in natural language and the AI answers in Japanese based on relevant handover records.  
All processing is local — no data is sent to external services.

## Tech Stack

| Component | Details |
|---|---|
| LLM | `qwen2.5:14b` (via Ollama, configurable) |
| Embedding model | `nomic-embed-text` (via Ollama) |
| Re-ranker | `BAAI/bge-reranker-v2-m3` (HuggingFace, multilingual with Japanese support) |
| RAG framework | LangChain (`langchain`, `langchain-ollama`, `langchain-chroma`, `langchain-community`) |
| Vector DB | ChromaDB (local persistent storage) |
| Morphological analysis (BM25) | SudachiPy + sudachidict_full + neologdn |
| Backend | FastAPI + Uvicorn |
| Frontend | Vanilla HTML/CSS/JavaScript (no dependencies) |

## How Search Accuracy Is Improved

Instead of simple vector search, a 4-stage pipeline is used:

```
Question
 │
 ├─ BM25 Search (strong for exact terms: equipment names, fault codes)
 ├─ Vector Search (broad semantic similarity)
 │
 ↓
RRF Fusion (Reciprocal Rank Fusion) → Top N candidates (configurable, up to 40)
 │
 ↓
CrossEncoder Re-ranking (precise relevance scoring) → Top N (configurable, default 5)
 │
 ↓
qwen2.5:14b generates answer in Japanese
```

| Method | Benefit |
|---|---|
| **BM25 (keyword search)** | Precise matching for equipment names, fault codes |
| **Vector search** | Catches semantically similar records broadly |
| **RRF fusion** | Combines both rankings into a unified list |
| **CrossEncoder re-ranking** | Precisely scores relevance between question and each record |

## Prerequisites

- macOS / Linux
- Python 3.11+
- [Ollama](https://ollama.com/) installed with the following models pulled:

```bash
ollama pull qwen2.5:14b
ollama pull nomic-embed-text
```

> **Apple Silicon (arm64) note**  
> `sudachipy` and `neologdn` ship arm64 wheels and install without issues via `pip`.  
> `sudachidict_full` (~120 MB) runs a build step on first install that may take a few minutes.

> **Choosing a model size** — pick based on your available VRAM (GPU) or unified memory (Apple Silicon):
>
> | Model | Recommended memory | Notes |
> |---|---|---|
> | `qwen2.5:32b` | 24 GB+ | Highest accuracy |
> | `qwen2.5:14b` | 12 GB+ | Default (balanced accuracy / speed) |
> | `qwen2.5:7b` | 8 GB+ | Lightweight / fast |
>
> If you change the model, use the model selector in the Web UI header or the `/api/model` API. Changes are saved automatically to `config.json`.

## Setup & Launch

### 1. Get the repository

```bash
git clone https://github.com/your-org/handover-rag-r1.git ~/handover-rag-r1
```

> If you downloaded a ZIP from GitHub, extract it to `~/handover-rag-r1/`.

> **config.json**  
> `config.json` is auto-generated by the app and is not tracked by git. See `config.json.example` for the template.  
> If absent, the app starts with the default LLM model `qwen2.5:14b` and reasoning disabled.

### 2. Place the CSV/XLSX file

Place the handover note file (CSV or XLSX) in the **current directory** (`~/handover-rag-r1/`).

Files named `引き継ぎノート*.csv` or `引き継ぎノート*.xlsx` are detected automatically.

```
~/handover-rag-r1/
├── 引き継ぎノート2026.xlsx   ← place here (引き継ぎノート*.csv or *.xlsx)
├── ingest.py
└── ...
```

If multiple matching files exist, the most recently modified one is used automatically.

#### When no matching file is found

If no file matching `引き継ぎノート*` is found, the system falls back in this order:

1. **Cursor selection**: lists all `.csv` / `.xlsx` files in the current directory
   - Navigate with `↑` / `↓` (or `k` / `j`), press `Enter` to select, `q` or `Esc` to cancel
2. **Manual input**: if no candidates exist or selection is cancelled, you will be prompted to type a file path

#### Specifying a file directly with `--csv`

You can also specify any file directly:

```bash
python ingest.py --csv /path/to/your_data.xlsx
```

### 3. Start Ollama

```bash
ollama serve
```

### 4. Launch the app

```bash
cd ~/handover-rag-r1
bash start.sh
```

On the first run, the following steps execute automatically:

1. Create a Python virtual environment (`venv`)
2. Install dependencies (`requirements.txt`)
3. Download the re-ranker model (`BAAI/bge-reranker-v2-m3`) — first run only, several hundred MB
4. Ingest the CSV/XLSX into ChromaDB (takes a few minutes depending on record count)  
   ※ If no `引き継ぎノート*` file is found, a file selection screen appears in the terminal
5. Start the server

Subsequent runs reuse the existing index and start immediately.

### 5. Open in browser

```
http://localhost:8000
```

## How to Use

### AI Answer Mode

Type a question and press **"AI 回答"** (AI Answer). The system retrieves relevant handover records and the configured LLM model generates a Japanese answer grounded in those records.

**Example questions:**
- `ポンプに関する故障はどのようなものがありましたか？` (What pump failures were there?)
- `2024年6月の夜勤での異常報告を教えてください` (Night shift anomaly reports in June 2024?)
- `高圧洗浄ポンプのトラブル事例は？` (Trouble cases with the high-pressure washing pump?)
- `水位計の洗浄作業はいつ行われましたか？` (When was the water level gauge cleaned?)

**Shortcut:** `Ctrl + Enter` (Mac: `Cmd + Enter`) to submit

### Search-Only Mode

Press **"検索のみ"** (Search Only) to display matching records directly without invoking the LLM — useful for quickly browsing raw records.

### Adjusting Result Count

Use the dropdown to select **3 / 5 / 10 / 20 / 30 / 40** results. More results means the AI references a broader set of records when generating its answer.

## User Dictionary Management

SudachiPy morphological analysis is used as BM25 preprocessing.  
Register plant-specific terminology (e.g. `冷却水ポンプ`, `最終沈殿池`) so they are treated as single tokens.

### Open the dictionary management UI

Click the **"辞書管理"** link in the header, or navigate directly to:

```
http://localhost:8000/static/dict_admin.html
```

### Basic workflow

1. **Register terms** — use "新規追加" (Add) or CSV import
2. **Apply dictionary** — click "辞書を反映" (Rebuild) to build user.dic and reindex BM25
3. **Verify** — use the tokenization test area to confirm compound terms become single tokens

> **Important**: registering entries alone does not affect BM25. Always click "辞書を反映" after changes.

### CSV import format

Header row required. `surface` and `reading` are mandatory; other columns are optional.

```csv
surface,reading,pos,cost,normalized,enabled
最終沈殿池,サイシュウチンデンチ,名詞,固有名詞,一般,3000,,1
冷却水ポンプ,レイキャクスイポンプ,名詞,固有名詞,一般,3000,,1
最沈,サイチン,名詞,固有名詞,一般,5000,最終沈殿池,1
```

| Column | Description | Default |
|---|---|---|
| `surface` | Written form (as it appears in documents) | required |
| `reading` | Reading in katakana | required |
| `pos` | Part of speech (comma-separated, up to 3 parts) | `名詞,固有名詞,一般` |
| `cost` | Priority cost (lower = higher priority) | `5000` |
| `normalized` | Normalized form (abbreviation → full name) | empty (same as surface) |
| `enabled` | Enabled (1) / Disabled (0) | `1` |

### Split modes

Controlled by `sudachi_mode` in `config.json` (A / B / C, default C).

| Mode | Behavior |
|---|---|
| **C** (recommended) | Keeps compound words intact (`冷却水ポンプ` → 1 token) |
| B | Moderate splitting |
| A | Fine-grained splitting into minimal units |

The mode can also be changed from the dictionary management UI and is saved to config.json automatically.

### AI-assisted terminology extraction

From the dictionary management UI, you can ask the LLM to extract domain terminology from sampled handover notes:

1. Click "候補を自動抽出" (Extract candidates) — LLM inference takes a few to tens of seconds
2. Review the candidates and fill in readings
3. Check the entries you want to register and click "選択を登録" (Register selected)

The sample size is controlled by `suggest_sample_size` in `config.json` (default: 200).

## Re-ingesting CSV/XLSX

### Via the Web UI (recommended)

Once the server is running, open the **"DB管理" (DB Management)** panel in the browser, select a CSV/XLSX file, and use the buttons:

| Button | Action |
|---|---|
| **差分更新** (Incremental update) | Add only new rows while keeping existing DB |
| **全件再構築** (Full rebuild) | Delete existing DB and re-ingest all records |

Progress is shown in real time via a progress bar.

### Via the command line

**Full rebuild** (replacing the data file entirely):

```bash
rm -rf ~/handover-rag-r1/chroma_db
bash start.sh
```

**Incremental update** (new rows appended at the end of the existing file):

```bash
cd ~/handover-rag-r1
source venv/bin/activate
python ingest.py                          # auto-detect or interactive selection
python ingest.py --csv path/to/file.xlsx  # specify a file directly
python ingest.py --rebuild                # full rebuild without deleting chroma_db manually
curl -X POST http://localhost:8000/api/reload-index
```

## CSV Format

The CSV has a title row, a header row, and data rows (referenced by 0-based index).

| Column index | Field | Description |
|---|---|---|
| 0 | (management) | Sequential number etc. (skipped during ingestion) |
| 1 | 故障 (Fault code) | Fault number (optional) |
| 2 | 記入年月日 (Date) | `YYYY/M/DD` format |
| 3 | 曜日 (Day of week) | Mon–Sun |
| 4 | 勤務 (Shift) | 日勤 (day) / 夜勤 (night) etc. |
| 5 | 種別 (Type) | 報告 (report) / 故障 (fault) / 処置 (action) etc. |
| 6 | 時刻 (Time) | 午前 / 午後 / HH:MM etc. |
| 7 | 内容 (Content) | Main text (primary search target) |
| 8 | 記入者 (Author) | Staff name |
